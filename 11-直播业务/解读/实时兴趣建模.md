## 一、背景与动机

### 1.1 直播场景下用户兴趣的"瞬时性"

在传统推荐场景（电商、图文Feed）中，用户兴趣相对**稳定**——一个喜欢数码产品的用户，未来几天大概率仍然对数码产品感兴趣。用户画像的更新频率可以是天级甚至周级。

但直播场景完全不同：

**用户兴趣在分钟级别剧烈波动。**

想象一个典型用户的晚间直播消费路径：
```
19:00 - 打开APP，随便看看 → 进入搞笑直播，看了3分钟
19:05 - 退出，刷Feed → 被一个游戏直播吸引，进去看了15分钟
19:20 - 游戏直播结束，退出 → 看到音乐直播，进去听了5分钟
19:25 - 觉得没意思 → 开始快速滑动，每个直播间只看几秒
19:30 - 发现一个美食直播 → 停下来认真看，看了20分钟
```

这个用户的"兴趣"在这30分钟内发生了多次变化。如果用离线画像（"该用户偏好游戏"），只能捕捉到他19:05~19:20的状态，其他时间都是错的。

### 1.2 传统离线兴趣的局限

```python
# 传统离线兴趣建模方式
offline_interest = {
    "user_123": {
        "category_preference": {"游戏": 0.4, "音乐": 0.3, "美食": 0.2, "搞笑": 0.1},
        "anchor_preference": ["anchor_A", "anchor_B", "anchor_C"],
        "active_time_pattern": "晚间20:00-23:00",
        "avg_watch_duration": "8分钟",
    }
}

"""
问题：
1. 时效性差：这是过去7天的统计，无法反映"此刻"的兴趣
2. 粒度太粗：只有品类级别的偏好，没有细分（如游戏-原神 vs 游戏-LOL）
3. 无法捕捉session内变化：用户进APP后的兴趣漂移完全丢失
4. 无法响应外部事件：比如某主播突然火了，用户的从众心理无法反映
"""
```

### 1.3 实时兴趣建模的核心目标

实时兴趣建模要解决的核心问题是：**在用户当前session内，根据最近几秒~几分钟的行为，动态捕捉其"此刻"的兴趣状态，并用这个状态指导后续推荐。**

具体来说：
- 用户刚刚连续进了3个游戏直播间 → 当前对游戏感兴趣 → 后续多推游戏
- 用户刚刚快速划过了5个直播间（每个<3秒） → 当前处于"无聊/探索"模式 → 推荐更有吸引力的内容或换品类
- 用户刚刚在某直播间打赏了100元 → 当前情绪高涨 → 可以推荐同类型的优质直播间

### 2.2 基于序列模型的实时兴趣编码

实时兴趣建模的核心是将用户最近的行为序列编码为一个**动态兴趣向量**。主流方法是使用序列模型（RNN/Transformer）处理实时行为序列。

```python
class RealtimeInterestEncoder:
    """
    实时兴趣编码器
    
    输入：用户最近N个行为组成的序列
    每个行为 = (直播间embedding, 观看时长, 互动类型, 时间间隔)
    
    输出：一个动态兴趣向量 h_realtime
    
    核心设计选择：
    - RNN（GRU/LSTM）：经典方案，串行处理，有遗忘机制
    - Transformer：并行处理，注意力机制，更灵活
    - Target Attention：用候选直播间做query，更精准
    
    工业界最常用的是 Target Attention 方案
    """
    
    def __init__(self, d_model=64, n_heads=4):
        # 行为编码器
        self.behavior_encoder = BehaviorEncoder(d_model)
        # 时间编码器（行为发生的时间间隔）
        self.time_encoder = TimeEncoder(d_model)
        # 自注意力层（捕捉行为间的关系）
        self.self_attn = MultiHeadAttention(d_model, n_heads)
        # 目标注意力层（用候选直播间query）
        self.target_attn = TargetAttention(d_model)
    
    def encode(self, behavior_seq, target_room_emb):
        """
        behavior_seq: list of (room_emb, duration, interact_type, time_delta)
        target_room_emb: 当前候选直播间的embedding
        """
        # Step 1: 编码每个行为
        behavior_embs = []
        for b in behavior_seq:
            h_b = self.behavior_encoder(b.room_emb, b.duration, b.interact_type)
            h_t = self.time_encoder(b.time_delta)
            behavior_embs.append(h_b + h_t)  # 融合时间信息
        
        # Step 2: 自注意力（捕捉行为间的关联）
        # 例如：用户连续看了两个游戏直播，自注意力会强化"游戏"信号
        behavior_seq_tensor = stack(behavior_embs)  # [N, d_model]
        h_self_attn = self.self_attn(behavior_seq_tensor)
        
        # Step 3: 目标注意力（用候选直播间做query）
        # 核心思想：用户的历史兴趣是多维的，面对不同的候选直播间，
        # 应该激活不同的兴趣维度
        # 例如：面对游戏直播间，激活"游戏"相关的历史行为
        h_interest = self.target_attn(
            query=target_room_emb,
            keys=h_self_attn,
            values=h_self_attn
        )
        
        return h_interest  # 实时兴趣向量


class TargetAttention:
    """
    目标注意力机制（Target Attention）
    
    这是工业界实时兴趣建模最常用的方案。
    核心思想：用候选物品（target）去查询用户历史行为序列，
    找到与当前候选最相关的历史行为，加权求和得到兴趣表达。
    
    与Self-Attention的区别：
    - Self-Attention: 行为之间互相attend（N×N注意力矩阵）
    - Target-Attention: 用target去attend历史行为（1×N注意力向量）
    
    经典工作：
    - DIN (Deep Interest Network) - 阿里，2018
    - DIEN (Deep Interest Evolution Network) - 阿里，2019
    """
    
    def forward(self, query, keys, values):
        """
        query: [d_model] - 候选直播间的embedding
        keys: [N, d_model] - 历史行为的embedding
        values: [N, d_model] - 历史行为的value（可以和key相同）
        
        返回: [d_model] - 加权的兴趣表达
        """
        # 计算query与每个key的相似度
        scores = matmul(query, keys.T) / sqrt(d_model)  # [N]
        
        # 可选：加入时间衰减权重（越近的行为权重越高）
        time_weights = exp(-decay_rate * time_deltas)  # [N]
        scores = scores + log(time_weights)
        
        # softmax得到注意力权重
        attn_weights = softmax(scores)  # [N]
        
        # 加权求和
        output = matmul(attn_weights, values)  # [d_model]
        
        return output
```

### 2.3 在线学习（Online Learning）

实时兴趣建模的终极形态是**在线学习**——模型参数本身也在实时更新，而不仅仅是特征在更新。

```python
class OnlineLearningSystem:
    """
    在线学习系统
    
    离线训练 vs 在线学习的区别：
    
    离线训练：
    - 每天/每周用全量数据训练一次
    - 模型参数更新慢（天级）
    - 可以充分利用数据（多次epoch）
    - 但无法快速响应分布变化
    
    在线学习：
    - 每来一个新样本就更新一次（或攒一批小batch更新）
    - 模型参数实时更新（秒级~分钟级）
    - 只能看一遍数据（single pass）
    - 能快速响应分布变化，但容易过拟合噪声
    
    工业界常用方案：离线训练 + 在线微调
    - 基础模型：离线训练好的大模型
    - 在线微调：用最近的数据对模型最后几层做实时SGD
    """
    
    def __init__(self, base_model, online_layers):
        """
        base_model: 离线训练好的完整模型
        online_layers: 需要在线更新的层（通常是最后几层）
        """
        self.base_model = base_model
        self.online_layers = online_layers  # 如tower头、最后的MLP
        self.optimizer = SGD(self.online_layers.parameters(), lr=1e-3)
    
    def online_update(self, features, label):
        """
        每来一个新样本就更新一次
        
        关键技巧：
        1. 只更新部分层（如最后一层MLP），不动底层参数
        2. 用较小的学习率，避免过拟合单个样本
        3. 加入正则化，防止参数漂移太远
        4. 设置更新频率上限（如每秒最多更新N次）
        """
        pred = self.base_model.forward(features, online_layers=self.online_layers)
        loss = BCE(pred, label)
        loss.backward()
        self.optimizer.step()
        self.optimizer.zero_grad()
    
    def predict(self, features):
        """推理时同时使用离线参数和在线微调后的参数"""
        return self.base_model.forward(features, online_layers=self.online_layers)


class FollowTheRegularizedLeader(OnlineLearningSystem):
    """
    FTRL（Follow The Regularized Leader）优化器
    
    工业界在线学习最常用的优化器之一（Google的广告系统广泛使用）。
    
    核心思想：
    - 每个参数维度独立更新
    - 对稀疏特征友好（不常出现的特征也能快速学习）
    - 内置L1正则化（自动做特征选择，不重要的特征权重归零）
    
    更新公式：
    z_t = z_{t-1} + g_t - (1/eta) * (sqrt(n_t) - sqrt(n_{t-1})) * w_t
    w_{t+1} = 0 if |z_t| <= lambda_1
             = -(z_t - sign(z_t)*lambda_1) / ((lambda_2 + sqrt(n_t))/eta) otherwise
    
    其中：
    - g_t: 当前梯度
    - n_t: 历史梯度平方和
    - eta: 学习率
    - lambda_1: L1正则化系数
    - lambda_2: L2正则化系数
```

### 2.4 Session-Level兴趣建模

除了实时更新模型参数，另一种思路是在session内动态调整推荐策略，而不改变模型本身。

```python
class SessionInterestManager:
    """
    Session级兴趣管理
    
    核心思路：
    在用户一次APP打开的session内，维护一个"兴趣状态机"，
    根据用户行为实时调整推荐策略。
    
    与在线学习的区别：
    - 在线学习：更新模型参数（全局影响所有用户）
    - Session管理：调整推荐策略（只影响当前session）
    """
    
    def __init__(self):
        self.session_state = SessionState()
    
    def update_on_enter_room(self, room, watch_duration, interacted):
        """用户进房并退出后，更新session状态"""
        
        # 更新品类访问频次
        category = room.category
        self.session_state.category_visits[category] += 1
        
        # 更新观看深度
        if watch_duration > 60:  # 有效观看
            self.session_state.deep_categories.add(category)
            self.session_state.satisfaction_score += 1
        elif watch_duration < 5:  # 秒退
            self.session_state.skip_count += 1
            self.session_state.satisfaction_score -= 0.5
        
        # 更新探索-利用状态
        if self.session_state.skip_count > 3:
            # 连续秒退 → 切换到探索模式
            self.session_state.mode = "explore"
            self.session_state.skip_count = 0
        else:
            self.session_state.mode = "exploit"
    
    def get_ranking_boost(self, candidate_room):
        """根据session状态，给候选直播间调整排序分"""
        
        if self.session_state.mode == "exploit":
            # 利用模式：强化已验证的兴趣品类
            category = candidate_room.category
            if category in self.session_state.deep_categories:
                return 1.5  #  boosting: 同类别加分
            else:
                return 1.0
        
        else:  # explore mode
            # 探索模式：鼓励多样性
            category = candidate_room.category
            if category in self.session_state.category_visits:
                return 0.7  # 看过的品类降权
            else:
                return 1.3  # 新品类加分
    
    class SessionState:
        def __init__(self):
            self.category_visits = Counter()  # 各品类访问次数
            self.deep_categories = set()       # 有效观看的品类
            self.skip_count = 0                # 连续秒退次数
            self.satisfaction_score = 0        # 满足度打分
            self.mode = "exploit"              # 当前模式：exploit/explore
```

## 三、实验结果

### 3.1 不同实时兴趣建模方案的对比

| 方法 | 进房CTR | 平均观看时长 | 用户满意度 | 工程复杂度 |
|------|---------|-------------|-----------|-----------|
| 纯离线画像 | 0.08~0.10 | 3.5min | baseline | ⭐ |
| +准实时特征(分钟级) | 0.10~0.12 | 4.2min | +10% | ⭐⭐ |
| +实时行为序列(DIN) | 0.12~0.14 | 5.0min | +20% | ⭐⭐⭐ |
| +实时序列(DIEN/Transformer) | 0.13~0.15 | 5.5min | +25% | ⭐⭐⭐⭐ |
| +在线学习微调 | 0.14~0.16 | 6.0min | +30% | ⭐⭐⭐⭐⭐ |
| +Session策略调整 | 0.15~0.17 | 6.5min | +35% | ⭐⭐⭐⭐ |

### 3.2 关键数据洞察

- **实时特征的边际收益递减**：从离线→准实时→实时，每步的提升幅度大约15~25%。但从实时→在线学习，提升开始放缓（5~10%），说明实时特征已经捕捉了大部分短期信号。

- **时间衰减的重要性**：在Target Attention中加入时间衰减权重，比均匀权重提升约8%。衰减系数通常设为半衰期=3~5个行为（即5个行为前的权重是当前的一半）。

- **在线学习的稳定性问题**：直接在线学习全部参数容易"跑飞"（模型崩溃），只微调最后1~2层+加正则化是最稳定的方案。工业界通常还会设置"安全阈值"——在线参数不能偏离离线参数太远。

- **序列长度的影响**：行为序列长度N=20~50时效果最好。太短（N<5）信息不够，太长（N>100）引入噪声且增加计算成本。

## 四、懂哥点评

### 4.1 工业价值 ⭐⭐⭐⭐⭐

实时兴趣建模是直播推荐中**差异化竞争**的关键：

1. **用户体验直接提升**：用户在session内能明显感受到"这个APP很懂我"——刚看了游戏直播，后续就推荐更多游戏内容。这种体验是离线画像无法提供的。

2. **商业价值显著**：实时兴趣建模直接影响用户的停留时长和打赏意愿。实验数据表明，实时推荐策略下用户的打赏率提升20~30%。

3. **技术壁垒高**：秒级特征更新、在线学习、低延迟推理——这些都需要扎实的工程能力，是小团队难以快速复制的。

### 4.2 局限性

1. **工程成本极高**：
   - 秒级特征需要完整的流计算链路（Kafka → Flink → Redis → 在线推理）
   - 在线学习需要额外的训练基础设施和监控体系
   - 任何一环的延迟或故障都会影响最终效果

2. **噪声与过拟合风险**：
   - 用户可能只是"偶然"进了一个直播间（误触、好奇），不代表真正的兴趣
   - 实时模型可能对噪声行为过度反应，导致推荐抖动

3. **评估困难**：
   - 实时策略的效果评估需要在线A/B测试，离线指标（AUC等）和在线指标可能不一致
   - session级别的指标（总观看时长）和单步指标（每次进房是否满意）需要平衡

### 4.3 适用场景

- **最适用**：DAU大、用户活跃度高、内容多样性强的直播平台
- **部分适用**：中小型直播平台（可以先从准实时特征开始，逐步演进）
- **不适用**：用户打开频率极低的场景（如工具类APP的直播入口）、内容同质化严重的场景
