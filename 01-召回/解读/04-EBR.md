## 一、EBR 的定位

EBR 不是一个新的模型架构，而是一篇**工程实践论文**，描述 Facebook 如何把向量召回从学术原型落地到每天服务几十亿用户的生产系统。

它的价值在于：把双塔 + 向量检索这套方案在工业级规模下会遇到的所有工程问题，以及对应的解法，都系统地讲清楚了。

## 二、双塔模型设计

### 2.1 特征体系

**Query 塔（用户侧）输入**：
- 文本特征：搜索词的 character n-gram（类似 DSSM 的 Letter-trigram）
- 用户画像：性别、年龄、地区、语言
- 上下文：时间、设备类型、历史搜索

**Document 塔（物品/页面侧）输入**：
- 实体文本：页面名称、描述、类别
- 社交特征：点赞数、评论数、分享数（反映流行度）

**关键设计**：两侧特征不对称，用户侧更多、更动态；物品侧更静态，可以离线全量预计算。

### 2.2 损失函数：Triplet Loss

$$\mathcal{L} = \max\left(0, \text{margin} + d(q, d^-) - d(q, d^+)\right)$$

其中 $d(\cdot, \cdot)$ 是欧氏距离（而非余弦，因为欧氏距离在 FAISS 中检索更高效）。

- $d^+$：正样本（用户点击的结果）
- $d^-$：负样本（曝光未点击）
- margin：间隔超参，通常设为 0.1

## 三、负样本的艺术（这是 EBR 最重要的贡献之一）

负样本的选择对双塔模型的质量有决定性影响：

### 3.1 随机负采样（Easy Negatives）
从全库随机采样：太简单，模型只需要学到"明显不相关"就能把 loss 推下去，没有学到细粒度判别能力。

### 3.2 In-batch 负样本
同一 batch 内其他 query 对应的 document 作为负样本：效率高，但有**假负样本问题**（同 batch 内可能有真正相关的 document）。

### 3.3 Hard Negatives（EBR 的核心贡献）
用**已训练模型的近邻检索**来挖掘难负样本：
- 用当前模型做 ANN 检索，取 Top-101 到 Top-500 的结果（排在前面但用户没点击）
- 这些是"模型觉得很相关但用户没选"的样本——正是模型需要学会区分的地方

训练流程变成迭代式（类似 Hard Negative Mining）：
```
Round 1: 随机负样本训练初始模型
Round 2: 用 Round 1 模型挖掘 Hard Negatives，再训练
Round 3: 迭代...
```

**实验结论**：加入 Hard Negatives 后，召回率提升 ~5%，相当显著。

### 3.4 混合负采样（最终方案）
- 1/3 随机负样本（保证模型不会 overfit 到 hard negatives）
- 2/3 hard negatives（提供细粒度判别信号）

## 四、FAISS 向量检索的工程考量

### 4.1 索引类型选择
Facebook Search 的候选集有数十亿规模，不同索引类型的 trade-off：

| 索引类型 | 精度 | 速度 | 内存 | 适用场景 |
|----------|------|------|------|---------|
| Flat（暴力搜索）| 100% | 极慢 | 大 | 仅用于评估 |
| IVF（倒排文件）| ~95% | 快 | 中 | 最常用 |
| IVF-PQ（量化压缩）| ~90% | 极快 | 极小 | 超大规模 |
| HNSW | ~99% | 快 | 大 | 精度要求高 |

**EBR 的选择**：IVF-PQ，在内存和召回率之间取平衡。

### 4.2 量化压缩（Product Quantization）
把 128 维向量分成 8 段，每段 16 维，每段单独量化到 256 个 code：
- 原始存储：128 × 4 bytes = 512 bytes/向量
- PQ 压缩后：8 bytes/向量（压缩 64x）
- 代价：检索精度从 ~99% 降到 ~90%

### 4.3 增量更新问题
物品向量需要定期更新（新内容加入、旧内容下线）。Facebook 的方案：
- **全量重建**：每天一次全量 ANN 索引重建（适合更新不频繁的场景）
- **增量更新**：新物品的向量实时插入（HNSW 支持增量插入，IVF 不支持）

## 五、量化评估指标

EBR 定义了两个专门用于评估向量召回的指标：

**KNN Recall@K**：
$$\text{KNN Recall@K} = \frac{|\text{ANN检索Top-K} \cap \text{暴力搜索Top-K}|}{K}$$
衡量 ANN 相对于精确搜索的损失。

**Full-Stack Recall@K**：
$$\text{Full-Stack Recall@K} = \frac{|\text{向量召回Top-K} \cap \text{点击物品}|}{|\text{点击物品}|}$$
衡量整个召回链路对用户真实点击的覆盖率（最终业务指标）。

## 六、线上效果

- 相比基于规则的召回（布尔检索）：Click +2%（搜索场景）
- 尤其在**语义召回**场景表现显著：能找到词面不同但语义相关的内容
- 冷启动新内容：上线即有向量，无需等待交互数据

## 七、懂哥点评

EBR 是工业级向量召回的"施工图"，研究生学的是模型，EBR 教的是**工程**。

几个值得反复咀嚼的点：

**关于负样本**：随机负样本给模型的信号太弱（"天鹅和萝卜的区别"），hard negatives 才是让模型学到细粒度判别能力的关键。但 hard negatives 太多会让模型 collapse（只学会区分 hard 的，忘了基本概念），所以要混合。

**关于 ANN 精度**：从 99% 降到 90% 看起来损失 9%，但因为候选集扩大了（从精确 Top-100 扩展到更多），实际 Full-Stack Recall 可能影响很小。精度损失不等于业务损失。

**关于增量更新**：这是工业落地的真实痛点。新内容怎么快速进入索引？实时性越高成本越大。不同业务场景的答案完全不同。
