# 融合公式设计 — 排序策略核心

> 懂哥解读系列 🐟 | 排序策略 · 多目标融合

---

## 一、背景动机

推荐系统的精排阶段，模型往往同时预估多个目标：**点击率（pctr）、完播率（pplay）、点赞率（plike）、评论率（pcomment）、转发率（pshare）**等。最终需要将这些分数融合为一个排序分（final score），用于决定展示顺序。

**核心矛盾：**
- 各目标的预估绝对值量纲差异巨大（pctr ~0.1, plike ~0.01, pcomment ~0.001）
- 业务期望多目标之间**可控权衡**，而非让模型自行决定
- 融合公式直接影响用户体验和业务指标，需要**可解释、可调控**

融合公式的设计哲学：**模型负责预估，公式负责决策**。

---

## 二、线性加法

### 公式

```
score = w1 * pctr + w2 * pplay + w3 * plike + w4 * pcomment + w5 * pshare
```

- 各目标的权重 `w_i` 由业务方手动设定或通过网格搜索确定。

### 优点
1. **简单直观**：公式完全可解释，改一个权重就能看到效果
2. **线性可加**：各目标贡献独立，不存在耦合效应
3. **工程友好**：计算开销极低，线上延迟无影响

### 缺点
1. **量纲敏感**：pctr ~0.1 天然主导排序，pcomment ~0.001 几乎被淹没
2. **权重调节成本高**：改一个权重可能同时影响多个目标，调参空间是连续的
3. **缺乏约束能力**：无法天然保证某个目标不低于某个阈值

### 适用场景
- 目标数量少（2-3个）且量纲相近
- 业务对精度要求不高，快速迭代阶段
- 作为 baseline 用于对比更复杂方案

---

## 三、指数乘法（几何加权）

### 公式

```
score = pctr^a * pplay^b * plike^c * pcomment^d * pshare^e
```

- 指数 `a, b, c, d, e` 控制各目标的相对重要性。

### 优点
1. **自动约束**：任何一项为 0 则总分为 0，天然避免"点击率高但完播率为 0"的内容排在前面
2. **无量纲问题**：指数运算自动归一化，不同量纲的目标都能发挥作用
3. **调节灵活**：增大某个指数即可提升该目标权重，效果比线性加法更显著

### 缺点
1. **过度惩罚**：某一项预估偏低就会被严重压制，可能误杀优质内容
2. **数值不稳定**：多个小数相乘容易下溢，需要 log 域计算
3. **缺乏可加性**：难以分析单个目标的边际贡献

### 为何能自动约束目标

核心在于**乘法结构的零传播特性**：
```
如果 p_i → 0，则 score = ∏ p_j^{w_j} → 0
```

这意味着：无论其他目标多高，只要有一个关键目标预估为零，该内容就会被自动降权。这在业务上等价于**硬约束**——"不允许在某个维度上完全失败"。

实践中常加入 **smoothing** 防止零值：
```
score = (pctr + ε)^a * (pplay + ε)^b * ...
```

## 四、带权指数加法（工业主流方案）

### 公式

这是目前**爱奇艺、快手、抖音**等工业界最常用的融合形式：

```
score = pctr^a * (w_like * plike^b + w_comment * pcomment^c + w_share * pshare^d)
```

或者更一般的形式：
```
score = pctr^a * Σ (w_i * p_i^{b_i})
```

### 为什么这种形式成为主流

1. **主目标 × 辅助目标**：pctr 作为主目标通过指数 a 控制，互动目标作为辅助加权和
2. **兼顾约束与灵活**：pctr 的零值仍会压制总分（乘法约束），但互动项之间是加法关系，不会过度惩罚
3. **业务对齐**：a 控制"流量效率"，w_i 控制"社区价值"，参数含义清晰

### 各家的具体实践

**快手（2020-2022 公开分享）：**
```
score = pctr^α * (w1 * pplay^β + w2 * plike^γ + w3 * pcomment^δ + w4 * pshare^ε)
```
- 强调 play（完播）的高权重，因为快手短视频场景完播是核心体验指标
- 互动项权重随社区治理需求动态调整

**抖音（公开技术分享）：**
```
score = pctr^a * pplay^b * (1 + w_like * plike + w_comment * pcomment + w_share * pshare)
```
- 使用 `1 + ...` 形式，保证互动为零时仍有一个 baseline 分数
- pctr 和 pplay 通过乘法保证基本的"可看性"约束

**爱奇艺（2021 技术博客）：**
```
score = pctr^a * (w_play * pplay + w_interact * (w_like * plike + w_comment * pcomment))
```
- 长视频场景下完播权重极高，互动项分层（基础互动 vs 深度互动）

## 五、归一化处理

### 问题

不同目标的预估值分布差异巨大：

| 目标 | 典型均值 | 典型 P99 |
|------|---------|---------|
| pctr | 0.08 | 0.35 |
| pplay | 0.45 | 0.95 |
| plike | 0.005 | 0.03 |
| pcomment | 0.001 | 0.008 |

直接使用原始预估值，pplay 会天然主导融合分数。

### 方案一：Min-Max 归一化

```
p_i_normalized = (p_i - min_i) / (max_i - min_i)
```

- 简单但受极端值影响大
- 线上实时统计 min/max 有延迟

### 方案二：分位数归一化（常用）

```
p_i_normalized = percentile_rank(p_i)  # 映射到 [0, 1]
```

- 将预估值转换为其在全体候选中的排名百分位
- 完全消除量纲差异
- 缺点：丢失绝对值信息，"两个都很好的内容"无法区分

### 方案三：排名替代预估（工业实践）

```
# 不融合预估值，而是融合排名
final_rank = f(rank_ctr, rank_play, rank_like, rank_comment)

# 典型形式
score = w1 * rank_ctr + w2 * rank_play + w3 * rank_like + ...
```

**优势：**
- 完全消除量纲和分布差异
- 排名是相对值，天然具有鲁棒性
- 对不同模型输出的兼容性极好

**劣势：**
- 排名是离散值，优化空间受限
- 无法利用绝对值信息（"点击率 0.5 vs 0.49" 排名可能相同）

### 方案四：Calibration + 融合

先用 isotonic regression 或 Platt scaling 对预估值做校准，使预估值与真实概率对齐，再进行融合。

## 六、自动搜参

### 6.1 网格搜索（Grid Search）

最朴素的方案：在权重空间中均匀采样。
- 适用于 2-3 个参数的低维空间
- 维度过高时计算量爆炸

### 6.2 CEM（Cross-Entropy Method）

**原理：**

CEM 是一种基于采样的黑箱优化方法，核心思想：

```
1. 初始化参数分布：θ ~ N(μ₀, σ₀²)
2. 从分布中采样 N 组参数 {θ₁, θ₂, ..., θ_N}
3. 对每组参数在线上做 A/B 测试，获得业务指标 R(θ_i)
4. 选取 Top-K（如 Top-20%）的参数
5. 用 Top-K 的参数更新分布：μ_new = mean(Top-K), σ_new = std(Top-K)
6. 重复步骤 2-5 直到收敛
```

**伪代码：**

```python
# CEM 自动搜参
mu = initial_params      # 初始均值
sigma = initial_std      # 初始标准差

for iteration in range(max_iter):
    # 采样 N 组参数
    samples = [np.random.normal(mu, sigma) for _ in range(N)]
    
    # 线上评估每组参数的业务指标
    rewards = [online_evaluate(s) for s in samples]
    
    # 选取 elite（Top-K）
    elite_indices = np.argsort(rewards)[-K:]
    elites = [samples[i] for i in elite_indices]
    
    # 更新分布
    mu = np.mean(elites, axis=0)
    sigma = np.std(elites, axis=0) + epsilon  # epsilon 防止方差坍缩
```

**实战效果：**
- 快手公开分享：CEM 相比人工调参，在相同迭代次数下业务指标提升 **3-5%**
- 优势：不需要梯度，对目标函数的连续性/可微性无要求
- 适合参数维度 5-20 的场景

### 6.3 贝叶斯优化（Bayesian Optimization）

**原理：**

```
1. 用高斯过程（GP）或 TPE 建立参数→指标的代理模型
2. 通过 acquisition function（如 EI/UCB）决定下一组要评估的参数
3. 评估后更新代理模型
4. 重复直到预算耗尽
```

**与 CEM 对比：**

| 维度 | CEM | 贝叶斯优化 |
|------|-----|-----------|
| 采样效率 | 中（需大样本量 N） | 高（每次迭代只需 1 组） |
| 并行性 | 好（N 组可同时线上评估） | 差（顺序依赖代理模型更新） |
| 维度适应性 | 中（10-20 维 OK） | 低（>10 维代理模型不准） |
| 实现复杂度 | 低 | 中 |
| 工业适用 | ★★★★★ | ★★★ |

**工业界的共识：CEM 更适合融合公式搜参**，原因是：
- 融合公式参数天然适合并行评估（同时上多组线上实验）
- 维度通常在 5-10 维，CEM 足够
- 实现简单，无需维护代理模型

## 七、个性化融合

### 思路

传统融合公式对所有用户使用相同的权重。但不同用户对不同目标的偏好天然不同：
- 有的用户爱点赞（互动型用户）
- 有的用户爱评论（社交型用户）
- 有的用户只是刷着看（消费型用户）

### 方案一：用户画像加权

```
w_like(u) = base_w_like * (1 + α * user_like_rate / avg_like_rate)
```

- 根据用户历史互动率动态调整权重
- 简单但有效，计算开销小

### 方案二：模型学习融合权重

```
# 将融合权重也参数化，由模型学习
score = f(pctr, pplay, plike, ..., user_embedding)
```

- 用一个轻量网络学习用户特定的融合策略
- 抖音在 2022 年公开的 ESMM 变体中使用了类似思路

### 方案三：Contextual Bandit

将融合权重选择建模为 contextual bandit 问题：
- state = user_embedding + content_embedding + context
- action = 选择一组融合权重
- reward = 用户后续行为反馈

### 挑战
- 个性化权重引入额外复杂度，调试困难
- 可能导致"信息茧房"——过度迎合用户偏好
- 需要充足的线上数据支撑

## 八、工业案例总结

### 快手（2021 KDD 分享）

- 融合公式：`pctr^α * Σ w_i * p_i^{β_i}`
- 参数搜搜：使用 CEM 在线自动搜参，每周迭代
- 关键经验：**完播率权重远高于其他互动项**，因为快手的"普惠"价值观要求内容被完整消费
- 特殊设计：对负反馈（不感兴趣、举报）设置**惩罚项**，直接从 score 中减去

### 抖音（2022 技术分享）

- 融合公式：`pctr^a * pplay^b * (1 + Σ w_i * p_i)`
- 关键经验：`1 + ...` 结构保证互动为零时仍有 baseline，避免过度依赖互动
- 个性化：融合权重与用户画像挂钩，高活跃用户更重视深度互动
- 自动搜参：使用贝叶斯优化 + 多臂老虎机混合策略

### 爱奇艺（2021 技术博客）

- 融合公式：分层设计，`pctr^a * (play_score + interact_score)`
- 关键经验：长视频场景完播不等于正反馈，需要区分"看完但无感"和"看完很喜欢"
- 特殊设计：引入**完播质量分**，综合完播进度、播放速度、回看行为

### 通用经验

1. **不要迷信复杂公式**：线性加法 + 好的归一化往往能打败复杂融合
2. **参数比结构重要**：同一个公式，好的参数 vs 差的参数，效果差距远大于不同公式之间的差距
3. **线上验证是唯一真理**：离线指标和线上效果的相关性不高，必须做 A/B
4. **定期重搜参数**：用户行为和生态都在变，融合参数需要持续迭代

## 九、懂哥点评 🐟

融合公式这件事，说白了就是一道**多目标优化题的简化版**。

学术上有多目标 Pareto 最优、约束优化等高大上的方法，但工业界选择了最朴素的答案——**加权融合 + 人工调参**。为什么？

1. **可解释性碾压一切**：老板问你"为什么点赞率掉了"，你能指着公式说"因为 w_like 从 5 调到了 3"。换成 Pareto front，你解释得清吗？

2. **调试友好**：融合公式出问题时，你可以逐个检查每个目标的预估和权重。黑箱方案一旦出 bug，定位成本极高。

3. **业务对齐**：产品经理能理解"提高评论权重"意味着什么，但理解不了"调整 Pareto 前沿的搜索方向"。

关于 CEM vs 贝叶斯优化：我的建议是**先用 CEM**。原因很简单——融合公式搜参天然适合并行实验，CEM 的并行优势能充分利用。贝叶斯优化的串行特性在线上场景反而是劣势。

关于归一化：**排名归一化是最稳的方案**。量纲差异是融合公式最大的坑，而排名直接消灭了量纲。虽然损失了一些绝对值信息，但在实际业务中，"谁比谁好"往往比"好多少"更重要。

最后一个忠告：**融合公式的天花板就是精排模型的天花板**。模型预估不准，公式调得再好也白搭。把 80% 的精力放在模型上，20% 放在融合公式上。
