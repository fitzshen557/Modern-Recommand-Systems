# P5 深度解读：用文本生成统一推荐任务

> **论文**：Recommendation as Language Processing (RLP): A Unified Pretrain, Personalized Prompt & Predict Paradigm (P5)  
> **来源**：Purdue University  
> **发表**：RecSys 2022  
> **论文链接**：https://arxiv.org/abs/2203.13366

---

## 一、P5 的核心问题

传统推荐系统有一个根本性的碎片化问题：

- **评分预测**：一套模型（矩阵分解系列）
- **序列推荐**：另一套模型（SASRec/BERT4Rec）
- **解释生成**：又一套模型（文本生成）
- **评论摘要**：再一套模型

每个任务都需要单独设计、训练、维护，知识无法跨任务共享。

**P5 的问题**：能不能用**一个统一的预训练语言模型**，通过不同的 Prompt，完成所有推荐相关任务？

---

## 二、P5 的设计框架

### 2.1 五类任务统一为文本生成

P5 把推荐系统的5类核心任务全部建模为 **seq2seq 文本生成**：

| 任务编号 | 任务类型 | 输入（Prompt）| 输出 |
|---------|---------|-------------|------|
| 1 | 评分预测 | "用户123对《哈利波特》的评分是？" | "5" |
| 2 | 序列推荐 | "用户123最近看了A、B、C，下一部最可能是？" | "《哈利波特》" |
| 3 | 直接推荐 | "给用户123推荐一本书" | "《哈利波特》" |
| 4 | 解释生成 | "为什么给用户123推荐《哈利波特》？" | "因为你之前喜欢..." |
| 5 | 评论总结 | "总结《哈利波特》的用户评价" | "大多数读者认为..." |

### 2.2 Personalized Prompt 设计

每个 Prompt 都包含：
- **用户 ID**：用特殊 token 表示，如 `<user_123>`
- **物品 ID**：同理，如 `<item_456>`
- **任务指令**：自然语言描述任务

示例 Prompt：
```
Prompt: 
"Given the following purchase history of user_123: 
[item_45, item_67, item_89, item_102], 
predict the next item that user_123 will purchase."

Output: "item_234"
```

### 2.3 模型骨干：T5

P5 使用 **T5（Text-to-Text Transfer Transformer）** 作为骨干：
- Encoder 处理 Prompt
- Decoder 自回归生成输出

用户 ID 和物品 ID 被加入 T5 的词表，对应的 Embedding 随机初始化后在微调时更新。

---

## 三、预训练与微调

**预训练**：用各类推荐数据集的多任务 Prompt 混合训练，目标是让模型学会"推荐语言"。

**微调**：在特定数据集上针对具体任务微调，指定 Prompt 模板。

**关键实验设现**：多任务预训练后，即使只微调特定任务，其他任务的性能也不会下降——这说明知识在任务间成功迁移了。

---

## 四、P5 的局限

| 局限 | 具体问题 |
|------|---------|
| **物品 ID 的语义缺失** | `<item_234>` 没有任何语义，模型只能靠协同过滤信号学习，冷启动无解 |
| **文本生成速度慢** | seq2seq 生成比 MLP 打分慢 10~100 倍，工业推断不现实 |
| **输出不可控** | 模型可能生成不在候选集中的物品 ID（幻觉问题） |
| **规模有限** | T5-Small/Base/Large，参数量远小于 GPT-3/LLaMA，知识存储有限 |

---

## 五、P5 的历史地位

P5 是 **LLM×推荐系统**这一方向的奠基性工作，意义在于：

1. **证明了统一框架的可行性**：一个模型真的可以做所有推荐任务
2. **开辟了 Prompt-based 推荐**：之后的 TALLRec、InstructRec 都在此基础上演化
3. **暴露了核心矛盾**：物品 ID 语义缺失 + 生成速度 → 催生了 TIGER（语义 ID）等后续工作

---

## 六、懂哥点评

P5 是一篇"开山之作"，论文本身在工业可用性上很弱（太慢、幻觉、ID无语义），但它提出的问题和框架极为正确：**推荐系统的碎片化是历史遗留问题，统一建模是正确方向**。

从 P5 到 OneRec（快手 2025），这条路走了3年，终于有了第一个工业落地案例。

读 P5 时要带着批判眼光：它的贡献是**提问**，而不是**答案**。真正的答案（如何在效率约束下实现统一生成式推荐）还在持续演化中。
