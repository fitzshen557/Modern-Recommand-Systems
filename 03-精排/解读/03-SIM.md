# SIM 深度解读：超长序列的两阶段检索

> **论文**：Search-based User Interest Modeling with Lifelong Sequential Behavior Data for Click-Through Rate Prediction  
> **来源**：阿里巴巴  
> **发表**：CIKM 2020  
> **论文链接**：https://arxiv.org/abs/2006.05639

---

## 一、长序列是精排的核心矛盾

电商平台上，活跃用户的历史行为可以达到**数千甚至数万**条。

但精排模型（DIN/DIEN）处理序列时：
- 注意力计算 $O(n)$
- GRU 序列计算 $O(n)$

**工业约束：精排延迟 <50ms，序列长度只能截断到 150~200 条。**

这造成了一个严重的信息损失：用户 18 个月前的购买行为可能对当前的推荐非常关键（比如买了吉他，现在大概率对琴弦、拨片有需求），但被截断丢弃了。

**SIM 的核心洞察**：不需要对所有历史行为做注意力，只需要找到**与 Target Item 相关的子序列**，再精细建模。

---

## 二、两阶段架构

```
用户全量行为序列（万级）
         ↓
  【第一阶段：General Search Unit (GSU)】
  快速检索，找到 Top-K 相关行为
         ↓
  相关行为子序列（K 条，K 一般为 50~200）
         ↓
  【第二阶段：Exact Search Unit (ESU)】
  精细注意力建模，等价于 DIN 或 DIEN
         ↓
  用户兴趣向量 → Concat → MLP → CTR
```

---

## 三、GSU：通用搜索单元

GSU 需要满足：**极快**（延迟 <5ms，不能成为精排瓶颈）、**覆盖率高**（不漏掉真正相关的行为）。

### 3.1 硬检索（Hard Search）——SIM 论文中使用

根据**行为物品的类目**做过滤：

$$\text{GSU}_{\text{hard}}(t, H) = \{h_i \in H : \text{category}(h_i) = \text{category}(t)\}$$

- Target Item 是"女士连衣裙" → 只保留历史中"女装类目"的行为
- 不需要任何模型计算，纯规则过滤，$O(n)$ 但常数极小
- 局限：依赖类目系统的质量，跨类目的相关行为会被过滤

### 3.2 软检索（Soft Search）——效果更好但更复杂

用**向量近似检索**：Target Item 的 embedding 作为 query，在历史行为的 embedding 空间中检索 Top-K：

$$\text{GSU}_{\text{soft}}(t, H) = \text{MIPS}(e_t, \{e_{h_i}\}_{i=1}^n, K)$$

- MIPS = Maximum Inner Product Search（最大内积搜索）
- 用 FAISS 等工具对每个用户的历史行为建索引（实时或离线）
- 优点：语义级相关性，不受类目边界约束
- 挑战：用户历史建索引的工程成本，实时性问题

---

## 四、ESU：精确搜索单元

ESU 接收 GSU 筛出的 $K$ 条相关行为，做精细的注意力建模——**直接复用 DIN 或多头注意力**。

因为序列长度从万级压缩到 $K$（通常 50~200），计算完全可接受。

同时加入**时间信息**：行为距今时间差 $\Delta t$ 也是重要特征，用于建模兴趣的时效性。

$$\text{ESU}(e_t, S_K) = \sum_{k=1}^{K} a(e_t, e_{s_k}, \Delta t_k) \cdot e_{s_k}$$

其中时间差 $\Delta t_k$ 被量化成 embedding 后作为辅助特征输入注意力计算。

## 五、线上工程实现

### 5.1 用户行为存储

全量历史行为（万级）不能放内存，需要存 KV 系统（如 Tair/Redis Cluster）：

```
Key:   user_id
Value: [(item_id, category_id, timestamp), ...]  ← 全量历史
```

检索时按类目过滤（Hard Search），或者预建 FAISS 索引（Soft Search）。

### 5.2 延迟拆解

| 阶段 | 操作 | 延迟 |
|------|------|------|
| GSU | KV 查询 + 类目过滤 | ~2ms |
| ESU | DIN 注意力（K=200） | ~5ms |
| 精排主体 | MLP + 特征 Lookup | ~20ms |
| 总计 | | ~27ms（可接受） |

## 六、实验效果

在阿里展示广告系统（淘宝首页）：
- 相比 DIN（截断到 200 条）：AUC **+0.009**
- 相比 DIEN（截断到 200 条）：AUC **+0.005**
- CTR 线上提升：约 **+7.1%**

历史序列越长的用户，SIM 的提升越显著——说明长期兴趣的确包含重要信息。

## 七、SIM 的演进

| 问题 | 改进 |
|------|------|
| GSU 硬检索依赖类目质量 | TWIN（快手）改用向量相似度做软检索，且预计算物品侧分数 |
| GSU 和 ESU 分开优化 | ETA（阿里）用 Hash LSH 做端到端可微的近似检索 |
| 序列建模仍用 DIN（无时序）| 加时间戳特征部分缓解，完整时序建模见 DIEN |

## 八、懂哥点评

SIM 是一个典型的**工程驱动的算法创新**：不是模型有多复杂，而是把一个清晰的工程问题（长序列 + 延迟约束）用架构上的巧思解决掉了。

"先检索、再建模"的思路其实在信息检索里早有先例（召回-精排），SIM 把这个思路搬到了序列建模里，非常自然。

实际落地要注意：
1. **GSU 召回率是上限**：如果 GSU 漏掉了关键行为，ESU 再精细也找不回来
2. **K 的选择**：太小漏掉信息，太大 ESU 变慢，通常 K=50~150 是甜点区
3. **硬检索 vs 软检索**：有类目系统的电商场景用硬检索简单高效；内容推荐（短视频/资讯）跨类目关联更复杂，需要软检索
