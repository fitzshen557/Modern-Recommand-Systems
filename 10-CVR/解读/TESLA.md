## 1. 背景与动机

### 从CVR到NetCVR：一个被忽视的问题

传统CVR建模回答的是"用户点击后会不会买"。但在电商真实场景中，**买了不代表最终转化完成**——用户可能退款。

```
真实电商漏斗：
曝光 → 点击 → 下单 → 确认收货（不退款）= 真正的转化

传统CVR只建模到"下单"，忽略了退款这一环
```

这就引出了一个更本质的指标：**净转化率（NetCVR）**

```
NetCVR = P(最终不退款 | 点击)
       = P(下单 | 点击) × P(不退款 | 下单)
       = CVR × (1 - 退款率)
```

### 为什么NetCVR建模很难？

NetCVR面临两个交织的难题：

**难题1：级联结构（Cascade）**

```
Click → Order → No-Refund

两个阶段的label不是同时可观测的：
- Order的label：下单后很快可知（小时级）
- No-Refund的label：需要等待很长的退款窗口（天级到周级）
```

**难题2：延迟反馈（Delayed Feedback）**

```
退款不是即时发生的：
- 有些用户第二天就退款
- 有些用户7天后退款
- 有些用户在确认收货后才退款（可能30天后）

问题：在训练时，我们看到的"未退款"样本，可能只是"还没退"
- 如果等太久才训练，label太滞后，模型更新不及时
- 如果不等就训练，label可能有噪声（假阴性：实际会退但还没退）
```

### 之前方法的不足

| 方法 | 思路 | 问题 |
|------|------|------|
| 固定等待窗口 | 等N天再取label训练 | 要么等太久导致label滞后，要么等太短导致噪声label |
| FNCW（Fake Negative Calibration） | 对"假阴性"样本加权 | 依赖对退款时间分布的假设，不够鲁棒 |
| 两阶段独立建模 | 分别建模CVR和退款率再相乘 | 忽略了两个阶段的特征交互和信息共享 |
| ESMM直接扩展 | 用CTCVR/CTR的思路 | 没有考虑退款信号的延迟特性 |

## 2. 方法详解

### 核心洞察

TESLA将NetCVR建模分解为**级联的两阶段问题**，并为每个阶段设计专门的延迟反馈处理机制。

```
TESLA的分解：

NetCVR(x) = P(order|x, click) × P(no_refund|x, order)
          = CVR(x)          × (1 - RefundRate(x))

Stage 1: CVR建模（下单率预估）—— 延迟较短
Stage 2: Refund Rate建模（退款率预估）—— 延迟较长
```

### 模型架构

```
                    共享特征（User/Item/Context Embedding）
                              |
                    +---------+---------+
                    |                   |
               Shared Bottom       Shared Bottom
                    |                   |
            +-------+-------+   +------+-----+------+
            |               |   |             |      |
        CVR Tower      Delay    Refund Tower  Delay  |
        (下单预估)    Weight    (退款预估)   Weight   |
            |           |         |           |      |
            |           |         |           |      |
         p(order)    w(t)    p(refund)     w'(t)   |
            |                     |                |
            +------- NetCVR = p(order) × (1-p(refund)) -------+
```

### 关键模块1：时间感知的延迟反馈处理

TESLA引入了**时间感知的样本权重**来处理延迟label问题。

```
核心思路：
对于"目前未退款"的样本，根据已经过去的天数，估计它是"真的不会退"还是"还没退"。

权重计算：
w(t) = P(退款已在t天内发生 | 最终会退款)

这个权重本质上是一个CDF（累积分布函数）：
- t=0时，w(0)≈0（刚下单，几乎不可能已经退了）
- t=7时，w(7)可能=0.7（7天内退了70%的退款）
- t=30时，w(30)≈0.95（30天后几乎全部退款都发生了）

直觉：
- 对于已经等了很久还没退的样本，我们更有信心它"真的不会退"→ 权重高
- 对于刚下单的样本，"没退"可能是假信号 → 权重低
```

### 关键模块2：序列学习机制

TESLA利用用户的历史行为序列来捕捉退款模式：

```
序列建模：
- 用户的历史退款模式（频率、金额、品类偏好）
- 商品的历史退款率
- 时序特征（下单时间、退款窗口期等）

使用类似DIN的注意力机制：
- 对用户行为序列中的每个历史订单计算attention score
- 重点关注与当前订单相似的历史订单的退款情况
```

### 关键模块3：级联训练策略

```
训练流程：

Phase 1: 先训练CVR模型
- 目标：P(order | click, x)
- 数据：所有点击样本，label = 是否下单
- 延迟：较短（下单通常在几小时内确认）

Phase 2: 训练Refund模型
- 目标：P(refund | order, x)
- 数据：所有下单样本，label = 是否退款
- 延迟：较长，需要时间感知的label校准

Phase 3: 联合微调
- 同时优化两个阶段
- 加入NetCVR的端到端loss
- 使用curriculum learning，逐步引入更长窗口的样本
```

### 关键公式（文字描述）

```
总Loss = L_CVR + L_Refund + L_NetCVR

L_CVR:
- 在点击样本上计算BCE loss
- label = 是否下单（延迟短，基本准确）

L_Refund:
- 在下单样本上计算加权BCE loss
- label = 是否退款（延迟长，用时间权重校准）
- 权重 = 时间感知的置信度 w(t)

L_NetCVR:
- 端到端的NetCVR预测loss
- pred_netcvr = pred_cvr × (1 - pred_refund)
- label = 是否最终转化（需要足够长的等待期才能得到准确label）

时间权重的学习：
- 可以用历史数据的退款时间分布来初始化
- 也可以在训练中作为可学习参数
- 参数化方式：logistic函数 / 指数衰减 / 分段线性
```

### 工程实现要点

```
# 时间权重计算的工程实现
def compute_time_weight(days_since_order, refund_cdf_params):
    """
    refund_cdf_params: 从历史数据拟合的退款CDF参数
    返回：该样本的"未退款"label的置信度
    """
    # 方法1：参数化CDF（如Weibull分布）
    cdf_value = 1 - exp(-(days_since_order / scale) ** shape)
    confidence = 1 - (1 - cdf_value)  # 越久没退，越有信心是真不ret

    # 方法2：分段统计（更鲁棒）
    # 按品类/价格段分别统计退款时间分布
    
    return confidence

# 训练时的label处理
def get_refund_label(sample, max_wait_days):
    days = sample.days_since_order
    if days >= max_wait_days:
        # 等够了，label确定
        return sample.is_refunded, weight=1.0
    else:
        # 没等够，label不确定
        if sample.is_refunded:
            return 1, weight=1.0  # 已经退了，label确定
        else:
            # 还没退，但可能后面会退
            confidence = compute_time_weight(days)
            return 0, weight=confidence  # 标签是0，但降低权重
```

## 3. 实验结果

### 离线实验

在阿里妈妈电商数据集上：

| 模型 | AUC (NetCVR) | 相对提升 |
|------|-------------|---------|
| ESMM (直接用于NetCVR) | 0.7102 | baseline |
| FNCW | 0.7189 | +1.23% |
| 两阶段独立模型 | 0.7234 | +1.86% |
| TESLA | **0.7356** | **+3.58%** |

### 在线实验

在阿里妈妈的电商推荐系统中：

| 指标 | 提升 |
|------|------|
| 净GMV | +3.2% |
| 退款率 | -1.8%（降低了低质量推荐） |
| 用户满意度 | 有正向趋势 |

关键发现：TESLA不仅提升了NetCVR预估准确性，还通过更好的退款预估间接降低了系统的退款率——因为模型学会了不推荐那些"容易退"的商品。

### 消融实验

| 变体 | AUC | 说明 |
|------|-----|------|
| TESLA完整版 | 0.7356 | - |
| 去掉时间感知权重 | 0.7278 | 时间权重贡献大 |
| 去掉序列学习 | 0.7298 | 序列信息有帮助 |
| 去掉级联训练 | 0.7312 | 训练策略有贡献 |
| 用固定等待窗口代替 | 0.7245 | 延迟反馈处理是核心 |

## 4. 懂哥点评

### 工业价值：⭐⭐⭐⭐⭐

TESLA解决了一个工业界长期忽略但非常重要的问题：

1. **商业价值巨大**：净转化率直接关系真实收入。推荐了100个订单但退了30个，和推荐了90个订单退了5个，后者的GMV更高。

2. **思路新颖**：将级联结构和延迟反馈结合起来，不是简单套用现有方案。

3. **时间感知权重**：这个设计非常实用，可以推广到其他延迟反馈场景（如广告归因）。

### 局限性

1. **数据要求高**：需要足够的退款样本才能学好退款模型。对于退款率极低的品类（如虚拟商品），模型可能退化。

2. **退款窗口定义**：不同品类、不同平台的退款政策差异大，需要针对性调参。

3. **label滞后**：即便有时间权重，NetCVR的训练label本质上还是比CTR/CVR滞后很多，模型更新速度慢于CTR模型。

4. **退款原因未建模**：退款有多种原因（质量问题、不想要了、尺码不对等），TESLA统一建模可能丢失信息。

5. **推理时效性**：退款模型的推理需要等待足够长的时间才能验证准确性，在线评估比较困难。

### 适用场景

- ✅ 电商平台（有明确退款机制的场景）
- ✅ 高客单价商品（退款影响更大）
- ✅ 服务类商品（退款周期长，更需要精确建模）
- ❌ 虚拟商品/数字内容（退款率极低或不适用）
- ❌ 退款政策不明确的场景
- ❌ 数据量不足的小众品类

### 未来方向

- 引入退款原因的多粒度建模
- 结合强化学习优化推荐-退款的全链路
- 将时间感知权重推广到更通用的延迟反馈框架
